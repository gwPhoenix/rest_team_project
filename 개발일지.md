## Day 2 (2026-06-16) — 핵심 기능 구현 (P0)

### 구현 기능

#### 1. 인증 시스템 (auth.js)
- **Firebase Auth**: 이메일/비밀번호, Google OAuth 지원
- **로컬 폴백**: Firebase 미설정 시 localStorage 기반 모의 인증으로 자동 전환
- Firebase 오류 코드 → 한국어 메시지 매핑 처리

#### 2. OpenAI 통합 분석 파이프라인 (openai.js)

```
경험 입력
  ↓
analyzeExperience()   → STAR 분석 (JSON 반환)
  ↓
transformCompetencies() → 직무 역량 변환
  ↓
analyzeCoverLetter()  → 자기소개서 피드백 (입력 시)
analyzeInterview()    → 면접 답변 분석 (입력 시)  [병렬 처리]
  ↓
generateQuestions()   → 예상 질문 + 꼬리 질문 생성
  ↓
generateMissions()    → 성장 미션 생성 (로컬 로직)
```

- 모든 AI 호출은 `response_format: json_object` 강제 적용으로 파싱 오류 방지
- `analyzeCoverLetter` / `analyzeInterview` 는 `Promise.all`로 병렬 실행하여 대기시간 단축

#### 3. 데이터 저장 (db.js)
- Firestore 우선, 실패 시 localStorage 자동 폴백
- 컬렉션 구조: `users/{uid}/analyses/{docId}`
- localStorage 최대 50건 유지

#### 4. 입력 페이지 (input.js)
- 직무 선택: 드롭다운 + 태그 버튼 UI (상호 동기화)
- 실시간 글자 수 카운터 (경험 1000자 / 자소서 2000자 / 면접 1500자)
- 임시 저장 (작성 중 자동 localStorage 저장)
- 분석 시작 전 유효성 검사 (직무·경험 필수, 30자 이상)

### 트러블슈팅

**문제 1**: `const` 선언 IIFE 모듈이 HTML 인라인 `onclick`에서 접근 불가  
→ `const`는 window 프로퍼티가 아니므로 인라인 핸들러의 `with(window)` 스코프에서 탐색 실패  
→ **해결**: 전역 노출이 필요한 모듈 변수를 `var`로 변경

**문제 2**: Firebase 초기화 시 placeholder 값("YOUR_API_KEY")으로도 `firebase.auth()`가 예외 없이 생성됨  
→ 실제 API 호출 시 오류 발생 → 로컬 폴백이 작동하지 않음  
→ **해결**: `FIREBASE_CONFIG.apiKey.includes("YOUR_")` 감지 후 초기화 자체를 건너뜀

## Day 3 추가5-7 (2026-06-17) — 네이버 실제 이메일 user_metadata 저장

### 문제
- Supabase Custom Provider는 UserInfo 비표준 필드(`naver_email`)를 `raw_user_meta_data`에 저장하지 않음
- Nav 드롭다운에 실제 이메일 대신 `naver_{id}@oauth.naver` 합성 이메일 표시

### 해결
OAuth 로그인 직후 `session.provider_token`(네이버 액세스 토큰)으로 직접 실제 이메일을 가져와 `user_metadata`에 저장

```js
// onAuthStateChange에서 SIGNED_IN + provider_token + naver_email 미저장 시 실행
const res = await fetch('https://openapi.naver.com/v1/nid/me', {
  headers: { Authorization: `Bearer ${session.provider_token}` },
})
const { response: profile } = await res.json()
await supabase.auth.updateUser({ data: { naver_email: profile.email } })
```

- `naver_email` 이미 저장된 경우 스킵 (재로그인 시 API 중복 호출 없음)
- `updateUser` 후 `refreshSession()`으로 user 상태 즉시 갱신

### 결과
| 위치 | 표시 값 |
|---|---|
| 대시보드 Email 컬럼 | `naver_{id}@oauth.naver` (관리자 식별용) |
| Nav 드롭다운 | 실제 네이버 이메일 (`naver_email`) |

## Day 3 추가5-4 (2026-06-17) — 네이버 로그인 Custom OAuth Provider로 전환

### 배경
- 기존 방식: Edge Function(`naver-auth`, `naver-callback`)으로 OAuth 처리 후 magic link로 세션 발급
- 문제: magic link 방식이 `auth.identities.provider`를 'email'로 고정 → 대시보드 표시 불가
- 해결: Supabase Custom OAuth Provider 기능으로 네이버를 정식 provider로 등록

### 변경 아키텍처

```
// 이전
window.location.href → naver-auth Edge Function → naver-callback → magic link → 세션

// 이후
supabase.auth.signInWithOAuth({ provider: 'custom:naver' })
  → Naver OAuth → Supabase /auth/v1/callback
  → naver-userinfo Edge Function (응답 정규화)
  → Supabase 유저 생성 (provider: 'custom:naver')
```

## Day 3 추가5-3 (2026-06-17) — Supabase 대시보드 네이버 provider 표시 수정

### 문제
- Supabase 대시보드 Authentication → Users의 Providers 컬럼이 네이버 유저도 'Email'로 표시
- 원인: Supabase magic link 방식으로 세션 발급 시 `auth.identities.provider`가 'email'로 고정됨
- `raw_app_meta_data.provider`는 수정됐지만 대시보드는 `auth.identities`를 읽음

### 해결 방법
`auth.identities.provider`를 직접 수정하는 SQL 함수 생성 후, Edge Function에서 호출

#### 1. Supabase SQL 에디터에서 함수 생성
```sql
CREATE OR REPLACE FUNCTION public.fix_naver_identity(user_id_param uuid)
RETURNS void
LANGUAGE sql
SECURITY DEFINER
SET search_path = auth, public
AS $$
  UPDATE auth.identities
  SET provider = 'naver'
  WHERE provider = 'email'
  AND user_id = user_id_param;
$$;
```
- `public` 스키마에 생성하되 `SECURITY DEFINER`로 `auth` 테이블 접근 권한 확보

#### 2. fix-naver-provider Edge Function 업데이트
```ts
// 기존: app_metadata만 업데이트
await admin.auth.admin.updateUserById(userId, { app_metadata: { provider: 'naver', providers: ['naver'] } })

// 추가: auth.identities.provider도 업데이트
await admin.rpc('fix_naver_identity', { user_id_param: userId })
```

// 변경 전

const isNaver  = rawEmail.endsWith('@oauth.naver')
const provider = isNaver ? 'naver' : (meta.provider ?? user?.app_metadata?.provider ?? 'email')
const email    = isNaver ? (meta.naver_email || rawEmail) : rawEmail

- fix 함수 실행 전/후 관계없이 항상 정확하게 '네이버 로그인' 뱃지 표시

## Day 3 추가2 (2026-06-17) — 구글·네이버 소셜 로그인 추가

### 구현 내용

#### 구글 로그인
- Supabase 내장 provider라 `signInWithOAuth({ provider: 'google' })` 한 줄로 구현
- Google Cloud Console OAuth 앱 등록 + Supabase 대시보드 설정으로 완료
- 소요 시간: 약 30분

#### 네이버 로그인 (Supabase Edge Function)
Supabase 미지원 provider라 OAuth 흐름을 직접 구현

| 함수 | 역할 |
|------|------|
| `naver-auth` | state 생성 → 쿠키 설정 → 네이버 OAuth 페이지 리디렉션 |
| `naver-callback` | code → 토큰 교환 → 유저 정보 조회 → Supabase 유저 생성 → 세션 발급 |

**유저 격리 전략**
- 네이버 고유 ID 기반 가상 이메일 사용: `naver_{id}@oauth.naver`
- 카카오·구글과 동일 이메일이어도 별개 유저로 DB 생성
- 실제 네이버 이메일은 `user_metadata.naver_email`에 보관

**CSRF 방지**
- `naver-auth`에서 `crypto.randomUUID()` state 생성 → HttpOnly 쿠키 설정
- `naver-callback`에서 쿠키 state와 URL state 비교 검증

#### 소셜 로그인 UI 개선 (Nav.jsx)
- 네이버 유저는 synthetic email 대신 실제 네이버 이메일 표시
- 드롭다운에 소셜 로그인 제공자 뱃지 표시 (네이버/카카오/구글 로그인)

### 트러블슈팅

**문제 1**: Edge Function JWT 인증 오류 (`UNAUTHORIZED_NO_AUTH_HEADER`)
→ 기본적으로 JWT 필요 → `--no-verify-jwt` 플래그로 재배포

**문제 2**: `profiles_username_key` 중복 제약 오류
→ `handle_new_user` 트리거가 username 중복 시 예외 미처리
→ `EXCEPTION WHEN unique_violation` 핸들러 추가로 해결

**문제 3**: Supabase admin `generate-link` REST 엔드포인트 404
→ 직접 REST 호출 대신 JS SDK(`supabase.auth.admin.generateLink`)로 전환

**문제 4**: 이미 가입된 유저 재로그인 시 `user_not_found`
→ `createUser` 실패 후 `listUsers`로 찾는 로직 제거
→ 'already' 에러면 바로 `generateLink`로 진행하도록 단순화

### 네이버 로그인 연동 절차 (재현 시 참고)
1. 네이버 개발자센터 앱 등록 → Client ID/Secret 발급
2. `supabase secrets set NAVER_CLIENT_ID=... NAVER_CLIENT_SECRET=... SITE_URL=...`
3. `supabase functions deploy naver-auth --no-verify-jwt`
4. `supabase functions deploy naver-callback --no-verify-jwt`
5. 네이버 앱 Callback URL → `https://{project-ref}.supabase.co/functions/v1/naver-callback`

## v2.1 (2026-06-17) — 네이버 로그인 추가 · provider 버그 수정

### 구현 내용

#### 네이버 OAuth (Edge Function 방식)
Supabase 내장 provider 목록에 네이버가 없어 Edge Function 2개로 직접 구현

| Function | 역할 |
|----------|------|
| `naver-auth` | 네이버 인증 URL로 리다이렉트 (state 쿠키 발급) |
| `naver-callback` | 인증 코드 수신 → 토큰 교환 → Supabase 유저 생성 → magic link 발급 |

- 합성 이메일 패턴: `naver_{naverUserId}@oauth.naver`
- `createUser` 시 `app_metadata: { provider: 'naver', providers: ['naver'] }` 설정

#### provider 'email' 덮어씌움 버그 수정
- **원인**: `generateLink({ type: 'magiclink' })` 링크를 브라우저가 검증할 때 Supabase 내부 OTP 플로우가 `app_metadata.provider`를 강제로 `'email'`로 덮어씀
- **시도 1**: `auth` 스키마 트리거 → 권한 오류 (42501: permission denied for schema auth)
- **해결**: `fix-naver-provider` Edge Function + 프론트엔드 자동 수정 조합

```
로그인 완료 (provider='email'로 오염)
  ↓
onAuthStateChange 감지
  ↓
email.endsWith('@oauth.naver') && app_metadata.provider === 'email' 조건 확인
  ↓
fix-naver-provider Edge Function 호출 (service role로 app_metadata 수정)
  ↓
refreshSession → provider = 'naver' 정상 반영
```

### 트러블슈팅

→ 호스팅 환경에서 `auth` 스키마 직접 접근 권한 없음  
→ **해결**: 프론트엔드 `onAuthStateChange` + `fix-naver-provider` Edge Function으로 우회
